# 15.4.1 Administrative Access to Network Devices
Network devices provides several administrative connection methods including:

- Console
- Telnet
- SSH
- HTTP
- HTTPS
# 15.4.2 Device Access Method Comparison
**Console Port**

- Console access involves connecting directly to the physical console port of a network device using a console cable and terminal emulation software, such as PuTTY or SecureCRT.
- This method provides out-of-band access to the device, allowing administrators to access the command-line interface (CLI) regardless of network connectivity issues.
- Console access is often used for initial device configuration and troubleshooting.

**Example**: You are setting up a new router in a data center. Since the router is not yet configured for network access, you connect directly to the console port using a console cable and PuTTY to perform the initial configuration.
**Telnet**

- Telnet is an in-band method to remotely establish a CLI session between a client and a server over a network.
- Cisco IOS includes both a Telnet server and Telnet client.
- Telnet transmits data, including usernames and passwords, in clear text, making it vulnerable to interception and unauthorized access.
- The best practice is to use SSH instead of Telnet.
- Telnet should only be used in a lab environment.

**Example**: In a controlled lab environment, you need to quickly test remote access to a switch. Since security is not a concern in this isolated environment, you use Telnet to connect to the switch.

**SSH**

- SSH (Secure Shell) is an in-band cryptographic network protocol that provides secure, encrypted communication between a client and a server.
- It offers similar functionality to Telnet but encrypts all data exchanged between the client and the server, including authentication credentials and command output.
- SSH is widely used for remote administration of network devices due to its strong security features, including public-key authentication and data integrity verification.

**Example**: You need to remotely manage a switch located in another office. To ensure security, you use SSH to securely access the CLI of the switch from your location.

**HTTP**

- Some routers and switches support web-based management connections allowing administrators access using HTTP (Hypertext Transfer Protocol).
- HTTP provides a graphical user interface (GUI) accessible through a web browser, allowing administrators to configure and monitor the device using a point-and-click interface.
- However, like Telnet, HTTP is not secure for administrative access as data is transmitted in clear text.

**Example**: You are quickly checking the status of a router's interface. For convenience, you use HTTP to access the router's web interface from within a secure internal network.

**HTTPS**

- HTTPS (Hypertext Transfer Protocol Secure) is the secure version of HTTP.
- It uses encryption (SSL/TLS) to protect data transmitted between the client and the server.
- Network devices with web-based management interfaces often support HTTPS for secure administrative access.
- HTTPS ensures that sensitive information, such as login credentials and configuration settings, remains confidential and cannot be intercepted by unauthorized parties.

**Example**: To securely configure and monitor a router from a remote location, you use HTTPS to access the router's web-based management interface, ensuring that all transmitted data is encrypted.

# 15.4.3 In-Band Management
When it comes to managing this information, there are two main paths the data can take:

- **In-Band Management**: This is when the information flows through the same channels as regular data on the organization's network or the internet.
- **Out-of-Band (OOB) Management**: This uses a separate, dedicated network solely for management purposes, with no regular traffic on it.

# 15.4.5 Out-of-Band Management
The preferred method for managing devices in a large network is to use Out-of-Band (OOB) management on a network that only administrators can access.
OOB management is generally suitable for large enterprise networks, providing high security and mitigating the risk of insecure management protocols on the production network. However, in some cases, in-band management may be preferable. For example, if a critical link between two core switches fails, an OOB network might not detect the failure. In such cases, secure in-band management is more appropriate.

Guidelines for OOB Management:

- Provide the highest level of security when using console ports and management interfaces.
- Mitigate the risk of passing insecure management protocols over the production network.

Guidelines for In-Band Management:

- Apply to devices that need to be managed or monitored.
- Use IPsec, SSH, or SSL when possible.
- Decide whether the management channel needs to be open at all times.

# 15.4.7 WebUI Configuration Example

Cisco IOS-XE routers and Catalyst switches come with a web-based management interface known as the WebUI or web GUI (Graphical User Interface). This tool allows administrators to configure and monitor the device using a web browser, offering a more visual approach to network management.

To configure a Cisco router or switch for WebUI access, you will need a **username** command configured and some **ip http** commands. In addition, securing WebUI with an access control list is recommended.

```
R1(config)# interface GigabitEthernet0/0/1
R1(config-if)# ip address 192.168.1.1 255.255.255.252 
R1(config-if)# no shutdown 
R1(config)# exit R1(config)# username admin privilege 15 secret cisco123 R1(config)# no ip http server 
R1(config)# ip http secure-server 
R1(config)# ip http authentication local 
R1(config)# access-list 10 permit 192.168.1.0 0.0.0.255 
R1(config)# ip http access-class ipv4 10
```

Briefly, these commands do the following:

- **username admin privilege 15 secret cisco123:** Creates a user named "admin" with full administrative access (privilege 15) and sets the password to "cisco123" (encrypted).
- **no ip http server:** Disables the standard HTTP server.
- **ip http secure-server:** Enables the HTTPS server.
- **ip http authentication local:** Uses the local username and password database for HTTP authentication.
- **access-list 10 permit 192.168.1.0 0.0.0.255:** Creates an ACL (numbered 10) permitting access from devices in the IP range 192.168.1.0 to 192.168.1.255.
- **ip http access-class ipv4 10:** Applies the ACL (numbered 10) to restrict HTTP access to the specified IP range.
IOS defaults enable both HTTP and HTTPS servers, allowing connections via http:// (unencrypted) or https:// (encrypted). For better security, Cisco recommends disabling the HTTP server. The HTTP server allows login via enable password, local username/password, or AAA settings. However, for security purposes recent IOS versions favor using local usernames for security.

To access all WebUI features, users must have a privilege level of 15. IOS defines two privilege levels by default: 0 (user mode) and 15 (privileged mode). Users with privilege 15 can use the CLI configuration mode, install software, erase configurations, and reload the router. The **username** command above with **privilege** set to **15** grants immediate access to privileged mode. Without privilege 15, users cannot perform advanced features in the WebUI.

The **ipv4** argument in the **ip http access-class** command is new in IOS 17. Although it is not required yet, if you enter only **ip http access-class 10**, you will receive the following message:

`This CLI will be deprecated soon, Please use new CLI 'ip http access-class ipv4 <1-99> | <access-list-name>'`

# 15.4.9 Cloud Device Management
For more sophisticated network management, Cisco currently offers two advanced cloud applications that provide comprehensive graphical interfaces:  Cisco Catalyst Center (formerly, Cisco DNA Center) and Cisco Meraki Dashboard. These tools not only support all available features of WebUI, but also allow for centralized management of multiple devices through a single user interface.

**Note**: Both Cisco Meraki and Cisco Catalyst Center require an active licensing agreement to access their respective management dashboards and perform configuration changes.

**Cisco Catalyst Center**

- **What It Is**: A powerful tool for managing Cisco Catalyst network devices, offering centralized management, automation, and advanced analytics.
- **Best For**: Large, complex networks requiring detailed oversight and control.
- **Features**: Centralized management, AI-driven automation, real-time analytics, and robust security.
- **Usage Scenario**: Ideal for enterprise networks with a mix of wired and wireless devices needing sophisticated management tools.
**Cisco Meraki Dashboard**

- **What It Is**: A cloud-based platform for managing network devices like switches, wireless access points, and security appliances.
- **Best For**: Small to medium-sized businesses or distributed networks.
- **Features**: User-friendly interface, real-time monitoring, zero-touch provisioning, and comprehensive analytics.
- **Usage Scenario**: Perfect for organizations looking for a simple, scalable, and remote-friendly network management solution.

### **Similarities**

Both Catalyst Center and Meraki Dashboard offer centralized management and real-time monitoring, providing essential insights into network performance. They are scalable, supporting networks of varying sizes, and offer tools to streamline network management.

### **Differences**

Catalyst Center is designed for larger, more complex networks that require detailed control and advanced management features, often deployed on-premises. In contrast, Meraki Dashboard is more accessible and easier to use, making it ideal for smaller or less complex environments, with the added benefit of cloud-based deployment for remote management.

### **Choosing the Right Tool**

Choose Catalyst Center if you manage a large enterprise network that needs comprehensive control and advanced features with on-premises deployment. If you need a straightforward, cloud-based solution for managing small to medium-sized or distributed networks, then Meraki Dashboard is the choice for you.

For more information, search for the **Cisco Catalyst Center** and **Cisco Meraki Dashboard** webpages online.
